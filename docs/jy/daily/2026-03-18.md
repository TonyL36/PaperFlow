## 已完成工作

### 1) 接口文档与功能文档整理

- 梳理现有 user-service 与 content-service 的接口输出，统一文档入口与阅读路径
- 按功能目录补充和对齐说明，减少“接口存在但缺少使用指引”的断层
- 校准本地文档生成脚本使用方式，明确常用参数与执行顺序

### 2) 本地联调与启动链路优化

- 复核本地启动脚本在多服务场景下的行为，确保前端、网关、业务服务可稳定联动
- 强化常见故障的排查顺序，覆盖端口冲突、环境变量污染、历史进程占用等高频问题
- 对齐开发环境下的默认访问地址与健康检查路径，降低新人上手成本

### 3) 前端稳定性与可用性改进

- 统一页面请求失败时的提示表现，避免出现“静默失败”导致误判
- 补齐关键页面的空态展示与回退路径，保证异常数据下仍可继续操作
- 校正部分导航与权限可见性细节，保证普通用户与管理员行为边界清晰

## 关键决策

- 继续采用“按功能分层文档”作为长期维护方式，减少大文档滚动维护负担
- 本地验证坚持“脚本优先 + 明确默认值”策略，优先消除环境差异带来的不确定性
- 前端错误处理保持“请求层统一封装 + 页面层可视化兜底”双层策略，平衡复用与可观测性
- 管理端能力仍以最小权限暴露为原则，避免将管理员入口扩散到普通用户路径

## 问题与修复

### 1) 文档入口分散导致信息检索效率低

- 问题：接口与功能说明分布在多个位置，排查问题时切换成本高
- 修复：统一按 by-feature 目录组织入口，并补充索引与关联说明

### 2) 本地联调时偶发请求指向错误地址

- 问题：历史环境变量残留导致前端请求目标与预期不一致
- 修复：在本地运行流程中明确 API 基地址优先级，优先使用脚本注入的安全默认值

### 3) 异常场景页面反馈不够直观

- 问题：部分页面在接口失败时仅显示通用错误，定位上下文不足
- 修复：统一异常提示组件与页面级兜底文案，提升问题可见性与可操作性
