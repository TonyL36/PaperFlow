## 梳理的缺陷与风险

### 1) 网关放行 /auth/** 导致 logout 不生效

- 现状：网关对 `/api/v1/auth/**` 直接放行，不解析 JWT，也不注入 `X-User-Id`
- 结果：`POST /api/v1/auth/logout` 到达 user-service 时缺少 `X-User-Id`，逻辑会“空操作”，refresh token 不会被吊销

处理：将 auth 放行范围收敛为 `register/login/refresh`，其余 auth 端点需要携带 `Authorization` 并通过网关注入用户头。

### 2) 前端报 500（SYS_HTTP_ERROR）但未必是后端 500

- 现状：本机环境变量/代理配置可能导致前端 dev server 的 API base 指向错误地址
- 结果：前端 fetch 失败时呈现 `HTTP 500 / SYS_HTTP_ERROR`，容易误判为后端崩了

处理：在一键启动脚本里显式注入 `VITE_API_BASE=http://localhost:3151`，避免本机环境变量污染。

### 3) Windows 下 jar 锁导致 spring-boot repackage 失败

- 现象：`spring-boot-maven-plugin:repackage ... Unable to rename ... .jar.original`
- 原因：已有 `java -jar ...` 进程占用 jar 文件，Windows 锁文件导致无法重命名

处理：一键脚本支持 `down` 清理，并新增 `-Force` 可按端口杀占用进程，降低卡死概率。

### 4) 缺少管理员用户管理闭环

- 现状：user-service 仅有 `/auth/*` 与 `/users/me`，缺少管理员用户管理 API
- 结果：无法授予/移除 ADMIN、无法禁用账号、无法吊销某用户 refresh token

处理：新增 `/api/v1/admin/users/**` 管理端点（需要 ADMIN）。
