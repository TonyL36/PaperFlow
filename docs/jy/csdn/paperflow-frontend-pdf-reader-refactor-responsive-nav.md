# 从巨型组件到可维护阅读器：我把 PDF 阅读页、响应式布局和顶部导航一起收口的两天实战总结

> 摘要：最近我集中重构了一版论文阅读页。表面上看，这轮像是在修几个前端样式问题，比如 PDF 页面不居中、划词向右溢出、缩放后左栏消失、顶部导航挤爆。但真正做下来会发现，它们背后其实是同一个工程问题：阅读页职责耦合太重，布局、渲染、交互和业务状态混在一个组件里，导致任何一个点出问题，都会牵连其他区域。本文记录我这两天是怎么把 `PDF` 阅读页从巨型组件拆成装配层、怎么修高倍率下的响应式布局、怎么把顶栏改成可收纳的三态导航，以及最后如何用测试和运行态验证把这一轮真正收口。
>
> 标签：React｜TypeScript｜pdf.js｜前端重构｜响应式布局｜工程复盘

很多前端项目里都会有一种“热点页面”。

一开始它只是一个功能页，后面不断加需求：

- 要支持 `PDF` 阅读；
- 要支持左侧缩略图；
- 要支持右侧 `AI` 问答；
- 要支持划词、引用、翻译；
- 还要兼顾高倍率缩放、答辩演示和日常维护。

最后的结果通常不是“功能越来越完整”，而是“这个页面越来越不敢动”。

我这次重构的阅读页，就是这种典型情况。

## 1. 这轮为什么必须重构，而不是继续补 patch

问题最早不是从“代码不好看”开始的，而是从用户体验直接炸出来的：

- 阅读页一个组件里同时管文档加载、主视图、缩略图、划词浮层、`AI` 对话和布局状态；
- 页面放大到 `150% / 175% / 200%` 后，左栏、主阅读区和右侧面板开始互相抢空间；
- 高倍率下划词有时会向右溢出，选区和文字对不齐；
- 顶部导航在登录态入口增多后容易撑爆一行；
- 本地联调时还出现过“服务在线，但页面白板”的情况。

如果继续按以前的方式修，通常会变成这样：

- 样式不对，加一条 `CSS`；
- 划词有问题，再补一个判断；
- 布局挤了，再塞一个 `if`；
- 导航爆了，再缩一点字号。

这种修法短期能止血，长期只会让组件越来越脆。

所以我最后的判断是：这轮必须先拆结构，再修体验。

## 2. 第一步不是调样式，而是把阅读页拆成装配层

原来的 `PaperPdfReaderPage` 本质上是个巨型组件，里面混着：

- `PDF` 文档加载；
- 主视图和缩略图联动；
- 文本选区与浮层；
- 引用片段管理；
- `AI` 消息流；
- 各种 `ref`、`useEffect` 和全局事件监听。

这种页面最危险的地方在于：你修改一个小逻辑，根本不知道会不会把另一个完全不相干的区域带坏。

我这次的拆法比较直接，把它改成了“装配层 + Hook + 展示组件”：

### 2.1 `usePdfDocument()`

负责：

- 加载和销毁 `PDF` 文档；
- 维护 `loading / error / doc` 状态。

这样页面本身不再直接碰 `pdf.js` 生命周期。

### 2.2 `usePdfViewport()`

负责：

- 当前页、可见页；
- 主阅读区和缩略图联动；
- 页面缩放和可用宽度适配；
- 渲染层需要的布局状态。

这一步把“阅读器视口”从页面本体里拆了出去。

### 2.3 `usePdfSelectionController()`

负责：

- 划词结果提取；
- 浮层出现和消失；
- 点击外部关闭；
- 滚动或缩放时隐藏浮层。

原来散落在页面里的 `window.getSelection()`、`mousedown`、`scroll` 监听，终于有了归属。

### 2.4 `usePaperReaderChat()`

负责：

- 右侧 `AI` 对话状态；
- 引用片段添加/删除；
- 翻译到对话；
- 发送消息和更新回复。

这样阅读页终于不需要自己维护一整套聊天状态机。

### 2.5 展示组件分离

我把 UI 明确拆成了两块：

- `PdfThumbnailRail`
- `PdfMainViewport`

页面只负责接线，不再亲自实现每一层细节。

这一步做完后，整个阅读页的复杂度一下就降下来了。后面再改布局、改划词、改 `AI` 面板，都可以在更小的范围里处理。

## 3. PDF 阅读器最难的，不是样式，而是坐标系

很多人第一次做 PDF 阅读页，都会先想当然地觉得：

- `canvas` 画出来；
- 透明文字层盖上去；
- 浏览器原生 selection 不就能用了。

但只要你真的开始做“划词引用”“片段翻译”“高倍率阅读”，问题就会出现。

这轮我后面确认到的核心结论是：

> 划词错位和向右溢出，本质上不是单纯的 CSS 问题，而是坐标系、文字宽度和可选范围一起出了偏差。

### 3.1 把 `zoom` 换成 `transform: scale(...)`

之前页面有一段逻辑依赖浏览器 `zoom`。

这个方案的问题是：

- 视觉上的缩放变了；
- 但 `canvas`、文字层和选区的几何关系并不总是同步稳定；
- 用户看到的结果就会像“页面壳在，正文位置不对”。

后来我改成了：

- 内部使用固定逻辑页宽；
- 外层通过 `transform: scale(...)` 控制显示缩放。

这样一来，阅读器内部坐标系会稳定很多。

### 3.2 固定逻辑页宽，先稳住内部几何

这轮我没有让页面每次跟着容器尺寸重新生成一套完全不同的几何，而是先固定一套逻辑宽度，再根据实际可用空间计算显示缩放。

可以理解成两层：

- 内部页坐标尽量稳定；
- 外部显示大小再做适配。

这一步的好处是：

- 不同窗口尺寸下，文字层和 `canvas` 更容易保持对齐；
- 后面修划词时，逻辑基线更稳定。

### 3.3 文字层不能只信浏览器自然排版

真正把“右边缘溢出”压下来的，是对文字层宽度的修正。

问题在于：浏览器算出来的文字宽度，不一定等于 `PDF` 原始文本的宽度。

特别是在这些场景里更容易漂：

- 字体回退；
- 高倍率缩放；
- 标点和英文混排；
- 浏览器自己的排版细节差异。

所以我最后做的是：

- 保留手写文字层；
- 根据 `PDF` 原始文本宽度和 DOM 实测宽度，动态算 `scaleX`；
- 靠近右边缘时，再按剩余可用宽度收紧，避免继续往外顶。

示意代码大概是这样：

```ts
const expectedWidth = item.width * logicalScale;
const measuredWidth = span.getBoundingClientRect().width;
const scaleX = expectedWidth / measuredWidth;

if (Number.isFinite(scaleX) && scaleX > 0.5 && scaleX < 2) {
  span.style.transform = `scaleX(${scaleX})`;
}
```

这一步不是“强行把字压扁”，而是尽量把 DOM 文本宽度拉回到更接近 `PDF` 原始几何的位置。

### 3.4 边缘水印过滤不能一刀切

另一个细节是：边缘竖排水印、日期之类的内容，原来有时也会被选进去。

但这件事也不能粗暴处理，因为：

- 你如果把所有边缘文本都去掉，很可能误伤页码；
- 有些正常正文也可能靠边。

所以最后只过滤满足这些条件的文本：

- 位于左右边缘；
- 方向近似竖排；
- 长度特征更像水印而不是正文。

这样可以尽量排除噪声，又不把正常内容一起误杀。

## 4. 高倍率下，布局优先级必须重新定义

结构拆开后，我碰到的第二类大坑其实是响应式布局。

阅读页一共有三块区域：

- 左侧缩略栏；
- 中间 `PDF` 主阅读区；
- 右侧 `AI` 对话面板。

在正常宽度下三块并排没问题，但一旦浏览器放大到 `150% / 175% / 200%`，问题就来了：

- 左栏没了，但左边空一大片；
- `PDF` 没有真正居中；
- 右侧面板还在挤压主阅读区；
- 再缩放回来时，缩略图恢复也可能不稳定。

这轮我最后把响应式策略收成了两个决策：

- `autoHideRail`
- `stackPanels`

简单说就是：

- 宽度先不够时，先隐藏左侧缩略栏；
- 再继续变窄时，不再硬撑双栏，而是把 `PDF` 和 `AI` 改成上下堆叠。

这套策略背后的核心顺序是：

> 先保主阅读，再收左栏，最后再决定 `AI` 面板怎么站位。

如果顺序反了，就很容易出现“辅助区域都还在，真正阅读区域反而被挤坏”的情况。

## 5. 阅读页头部和全局导航，本质上是同一个问题

后来我发现一个挺典型的现象：

阅读页主体快修好了，顶部又开始出问题。

### 5.1 阅读页头部

阅读页上方有这些操作：

- `论文阅读`
- 返回文章详情
- 隐藏缩略栏
- 自动隐藏提示

高倍率下如果继续强行一行排开，就会互相挤压。

所以我做了一个很实用的改动：

- 让通用页面壳支持 header / title row / actions 的扩展类名；
- 阅读页在窄宽度下切换成纵向堆叠头部。

这样头部终于不会再和主体区一起“挤成一团”。

### 5.2 顶部导航

全局导航的问题其实更典型。

登录态下入口一多，原来很容易：

- 一整行被撑爆；
- 所有入口都挤在一起；
- 高频和低频入口没有区分。

我最后把导航改成了三态：

- `full`：图标 + 文本完整展示；
- `compact`：核心入口更紧凑；
- `collapsed`：次级入口收进 `More`。

同时把入口分层：

- 常驻核心：`Feed`、`Viz`、`Pathfinder`
- 登录态高频入口尽量常驻：`Favorites`、`Messages`
- 次级或管理入口进 `More`

这一步的关键不在“做了个菜单”，而在于终于承认一件事：

> 顶部导航不可能在所有宽度下都完整平铺，必须有收纳策略。

## 6. 这轮怎么验证不是“看起来修好了”

如果只是手动点一遍，很多结构性回归其实看不出来。

所以这轮我补了几类测试：

### 6.1 结构合同测试

验证重构后这些边界是不是还在：

- 阅读页是不是已经变成装配层；
- `usePdfDocument()`、`usePdfViewport()`、`usePdfSelectionController()`、`usePaperReaderChat()` 是否都已经接入；
- 原本页面里那些不该继续存在的状态和副作用是否已经移走。

### 6.2 响应式合同测试

验证这些规则有没有被后续改动带坏：

- 左栏自动隐藏；
- 面板堆叠；
- 阅读页头部高倍率响应；
- 顶部导航三态切换。

### 6.3 文字层测试

重点验证：

- 文本宽度修正逻辑；
- 边缘过滤逻辑；
- 文字层基础接线是否正常。

### 6.4 本地运行态验证

除了测试，我还固定了一条稳定的联调路径：

- 本地开发服务器固定 host；
- 前后端联调走稳定入口；
- 首页、帖子流、阅读页三个关键路径都要实际打开看。

这一步很重要，因为阅读器这类页面经常会出现“测试都过了，但真实页面一开就不对”的情况。

## 7. 这轮最重要的工程判断

回头看，这两天最有价值的不是“修了多少 bug”，而是做对了几件事：

### 7.1 热点页面必须拆层

阅读页这种高复杂度页面，必须坚持：

- 页面装配层；
- 功能 Hook；
- 展示组件分离。

不然需求越叠越多，最后谁都不敢改。

### 7.2 PDF 选区问题不能只靠 CSS

高亮对不齐、划词溢出这类问题，最终还是要回到：

- 坐标系；
- 原始文字宽度；
- 文字层可选范围。

单调样式最多改善观感，解决不了根问题。

### 7.3 响应式不是“缩一缩就行”

像阅读页这种多区域页面，必须明确布局优先级。

在我的场景里，优先级就是：

- 主阅读区优先；
- 左栏其次；
- 右侧辅助面板最后。

### 7.4 导航必须有层级和收纳

顶栏不能假设“所有入口永远都能塞进一行”。  
尤其是有登录态、管理态、常用入口共存的时候，更要提前设计 `More` 或折叠策略。

## 8. 最后的结果

这轮做完后，阅读页和导航至少达到了一个我能接受的状态：

- 阅读页从巨型组件变成了可维护的装配层；
- `PDF` 渲染、划词、缩略图、`AI` 对话的职责边界更清晰；
- 高倍率下的左栏、主区、右侧面板站位比之前稳定很多；
- 顶部导航不再只能硬挤一行；
- 结构测试、响应式测试和运行态验证路径都补上了。

当然，这不代表阅读器从此“彻底完美”了。

但至少现在它已经从一种“能跑但很脆”的状态，变成了“还能继续安全往下改”的状态。

## 9. 如果你也在做类似阅读页，我的建议是

- 不要等页面彻底失控了才拆，热点页面越早分层越好；
- 遇到 `PDF` 划词问题，先怀疑几何和宽度，不要先怪颜色和圆角；
- 多区域页面一定要明确布局优先级，不要让每个板块都争空间；
- 响应式不要只看 `CSS`，也要看组件状态和渲染接线；
- 手动验证很重要，但最好还是补几条结构合同测试，不然下一次重构还会反复踩坑。

## 一句话总结

这两天我做的，本质上不是“把 PDF 阅读页修顺眼一点”，而是把一个已经开始失控的复杂页面，重新拉回到可以继续维护、继续演进的工程状态。
