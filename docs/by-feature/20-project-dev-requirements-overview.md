# 20 项目开发要求总览（Project Dev Requirements Overview）

## 20.1 文档目标

本文用于给新接手同学、并行开发成员、测试与交付同学提供统一的“开发要求总览”，解决以下问题：

- 项目要做什么（业务目标与当前交付边界）
- 需求分层怎么落地（愿景 → 能力 → 功能 → 接口 → 实现）
- 系统由哪些模块构成、模块如何协作
- 关键端到端流程如何跑通与验收
- 本地与环境部署如何执行
- 当前 Done / Todo 与交接清单是什么

适用范围：当前仓库内的前端 SPA、API 网关、用户服务、内容服务与配套脚本/文档链路。

## 20.2 项目目标（Goals）

结合仓库现状，当前阶段目标可概括为“可联调、可演示、可迭代”的科研内容平台基础底座：

- 面向 AI 研究者提供统一入口，承载内容浏览、评论互动、用户资料与基础管理能力
- 通过统一 API 网关沉淀通用治理能力（鉴权、限流、错误归一化、路由转发）
- 以前后端解耦方式支持双模式开发（前端 Mock 联调 + 真实网关联调）
- 为后续 5-Agent 能力接入预留统一入口与治理边界（本阶段不实现 Agent 业务本体）

当前交付边界不包含复杂推荐、完整知识图谱构建、生产级多租户与 Agent 深度业务能力。

## 20.3 需求分层（Requirements Layers）

建议以五层需求模型维护开发要求，避免“只见接口不见目标”：

### L1 愿景层（Why）

- 降低科研文献信息过载与知识管理碎片化
- 提供从内容获取到互动沉淀的基础闭环

### L2 业务能力层（What）

- 用户身份体系：注册、登录、刷新、资料维护、绑定扩展
- 内容体系：帖子展示、详情阅读、评论发布、管理员审核
- 平台治理：请求追踪、鉴权透传、限流、防御性错误格式

### L3 功能与页面层（Feature）

- 前端页面：登录、帖子列表/详情、收藏/足迹、管理员用户/评论、可视化页面
- 后端功能：网关治理、用户资料与认证、内容发布与审核
- 文档能力：接口文档生成、按功能文档沉淀、脚本化运行说明

### L4 接口与契约层（API Contract）

- 统一前缀：`/api/v1`
- 统一响应：`requestId + data/error`
- 统一错误码语义与状态码映射
- 鉴权方案：Access Token（JWT）+ Refresh Cookie

### L5 实现与运行层（Implementation）

- 前端：React + Vite（`apps/paperflow-web`）
- 后端：Spring Boot 微服务（gateway / user / content）
- 数据：PostgreSQL（用户与内容）
- 运维：Docker Compose + PowerShell/Bash 脚本

## 20.4 架构总览（Architecture）

当前采用“表现层 - 业务层 - 数据层”三层映射：

- 表现层：React SPA，承载用户交互、状态管理与页面编排
- 业务层：API Gateway + User Service + Content Service
- 数据层：PostgreSQL（可扩展外部存储由网关治理接入）

关键架构要求：

- 所有外部请求优先进入网关，避免客户端直连业务服务
- 通用能力（鉴权、限流、错误归一化）集中在网关收口
- 业务服务保持职责单一，减少跨服务耦合
- 文档与脚本必须可复现，保障新成员在最短路径跑通

## 20.5 模块职责（Modules）

### 1) 前端（`apps/paperflow-web`）

- 负责页面呈现、用户交互与请求编排
- 提供 Mock/真实网关两种联调模式
- 统一错误态、空态与权限可见性处理

### 2) API 网关（`backend/services/api-gateway`）

- 路由转发到用户/内容服务
- 注入 requestId、处理 JWT、执行限流
- 归一化非规范错误，输出统一响应信封

### 3) 用户服务（`backend/services/user-service`）

- 认证链路：注册、登录、刷新、注销
- 用户资料与状态管理
- 外部绑定能力预留（如 OAuth/第三方绑定）

### 4) 内容服务（`backend/services/content-service`）

- 帖子查询（列表/详情）
- 评论创建与展示
- 管理端评论审核与状态流转
- 每日内容生成与演示数据能力

### 5) 脚本与部署（`scripts` / `docker`）

- 本地启动脚本、部署脚本、文档生成脚本
- 多环境 compose 组织（dev/test/prod）
- 环境变量样例与本地私有配置隔离

## 20.6 关键流程（Flows）

当前最重要的验收流程建议固定为以下 6 条主链路：

1. 用户注册/登录/刷新/获取资料  
2. 帖子列表与详情浏览  
3. 登录后评论发布与展示  
4. 管理员审核评论（通过/驳回）  
5. 内容服务每日帖子生成（定时/幂等）  
6. 前端 Mock 模式切换到真实网关模式

流程约束：

- 受保护接口必须依赖网关校验并透传用户身份
- 关键写操作需要满足幂等与可审计要求
- 所有失败路径必须返回统一错误结构并可定位 requestId

## 20.7 部署与运行要求（Deploy / Run）

### 本地开发建议顺序

1. 先跑前端 Mock，验证页面与交互  
2. 再启动网关与业务服务，切到真实模式联调  
3. 执行基础接口 smoke 与关键页面回归  

### 常用命令（Windows）

```powershell
cd F:\Gitee\PaperFlow\PaperFlow

# 一键本地启动（可选 build）
.\scripts\run-local.bat up
.\scripts\run-local.bat up build

# 前端 Mock
.\scripts\run-spa-mock.ps1

# 前端接真实网关
.\scripts\run-spa.ps1

# 多环境部署
.\scripts\deploy.ps1 -Env dev
```

### 配置要求

- 敏感配置放在 `scripts/env/local.env.bat`（已忽略提交）
- 端口、数据库、JWT、邮件、OAuth 等参数必须通过环境变量注入
- 团队协作时优先共享 sample 配置，不共享真实密钥

## 20.9 交接清单（Handoff）

建议交接按“代码、配置、流程、风险、计划”五件套执行：

1. 代码入口
   - 前端：`apps/paperflow-web`
   - 网关：`backend/services/api-gateway`
   - 用户：`backend/services/user-service`
   - 内容：`backend/services/content-service`

2. 运行入口
   - 本地一键：`scripts/run-local.bat`
   - 前端联调：`scripts/run-spa-mock.ps1`、`scripts/run-spa.ps1`
   - 部署：`scripts/deploy.ps1`

3. 文档入口
   - 功能索引：`docs/by-feature/00-index.md`
   - 流程总览：`docs/api-flow.md`
   - 设计规范：`docs/api-design-spec.md`
   - 变更记录：`docs/daily/*`

4. 必讲风险
   - 本地环境变量残留导致接口目标错误
   - 端口占用与历史进程未清理导致联调失败
   - 前端模式（Mock/真实）切换时的基地址不一致

5. 首周执行计划（接手人）
   - Day 1：本地全链路跑通 + 文档索引通读
   - Day 2：完成一条后端接口改动并更新对应文档
   - Day 3：补一条自动化验证或故障手册案例
